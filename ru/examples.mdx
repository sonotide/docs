## Зачем нужны эти примеры?

Консольные приложения, которые компилируются вместе с Sonotide, — это не просто декоративные демо. Они являются изоляционными интеграционными тестами конкретных подсистем:

  <Card title="List Devices" icon="list">
    Изучение поиска устройств через `runtime`.
  </Card>
  <Card title="Render Silence" icon="volume-low">

  </Card>
  <Card title="Playback Session" icon="play">
    Высокоуровневая сессия плеера.
  </Card>
</CardGroup>

Воспринимайте их как эталонный, рабочий код, который показывает минимально возможную точку входа в слой фреймворка.

## `sonotide_list_devices.exe`

Этот пример полезен, когда нужно убедиться, что Sonotide видит Windows
аудиостек и возвращает стабильные метаданные endpoint-ов.

### Что делает пример

1. Создаёт [`runtime`](/ru/api#api-runtime).
2. Вызывает `enumerate_devices(device_direction::render)`.
3. Печатает `id`, `friendly_name` и флаг устройства по умолчанию.

### Ключевой код

```cpp
auto runtime_result = sonotide::runtime::create();
if (!runtime_result) {
    std::cerr << runtime_result.error().message << '\n';
    return 1;
}

sonotide::runtime audio_runtime = std::move(runtime_result.value());
auto devices_result = audio_runtime.enumerate_devices(sonotide::device_direction::render);
if (!devices_result) {
    std::cerr << devices_result.error().message << '\n';
    return 1;
}

for (const auto& device : devices_result.value()) {
    std::cout << device.id << " | " << device.friendly_name;
    if (device.is_default_multimedia) {
        std::cout << " | default multimedia";
    }
    std::cout << '\n';
}
```

### Почему он устроен именно так

- [`runtime::create()`](/ru/api#api-runtime) задаёт границу фреймворка. Если
  он не поднялся, пример сразу печатает структурную ошибку и завершается.
- [`device_info`](/ru/api#api-device-model) уже достаточно для application code.
  Пример не лезет в COM-интерфейсы.
- Печать default-флага делает пример полезным не только как smoke test, но и
  как заготовку для UI выбора устройства.

### Как адаптировать под свой код

Этот шаблон подходит, если вам нужны:

- выбор устройства вывода в настройках;
- диагностическая команда, которая печатает endpoint-ы;
- первый шаг миграции с прямого `IMMDeviceEnumerator`.

### Запуск

```bat
build\msvc-x64-debug\examples\Debug\sonotide_list_devices.exe
```

## `sonotide_render_silence.exe`

Этот пример нужен, когда вы хотите проверить output path без собственного
декодера, DSP-цепочки или генератора тестового сигнала.

### Что делает пример

1. Создаёт [`runtime`](/ru/api#api-runtime).
2. Объявляет минимальный [`render_callback`](/ru/api#api-streams).
3. Открывает [`render_stream`](/ru/api#api-streams) с конфигурацией по умолчанию.
4. Запускает поток и завершается успешно, если критический render path поднялся.

### Ключевой код

```cpp
class silence_callback final : public sonotide::render_callback {
public:
    sonotide::result<void> on_render(
        sonotide::audio_buffer_view buffer,
        sonotide::stream_timestamp) override {
        std::fill(buffer.bytes.begin(), buffer.bytes.end(), std::byte{0});
        return sonotide::result<void>::success();
    }
};

sonotide::render_stream_config config;
auto stream_result = audio_runtime.open_render_stream(config, callback);
if (!stream_result) {
    std::cerr << stream_result.error().message << '\n';
    return 1;
}

auto start_result = stream_result.value().start();
if (!start_result) {
    std::cerr << start_result.error().message << '\n';
    return 1;
}
```

### Почему он устроен именно так

- Callback пишет нули в [`audio_buffer_view`](/ru/api#api-audio-format), поэтому
  пример детерминированный и не создаёт слышимого сигнала.
- [`render_stream_config`](/ru/api#api-streams) намеренно остаётся в дефолтном
  виде, чтобы пример проверял именно стандартный путь фреймворка.
- Успешный `start()` подтверждает, что выбор устройства, negotiation формата,
  запуск worker thread и callback path уже работают корректно.

### Как адаптировать под свой код

Это правильная отправная точка, если приложение уже владеет декодированным PCM.
Вместо заполнения нулями вставьте копирование ваших подготовленных кадров в
`buffer.bytes`.

### Запуск

```bat
build\msvc-x64-debug\examples\Debug\sonotide_render_silence.exe
```

## `sonotide_playback_session.exe`

Этот пример показывает high-level playback layer в реальной работе: загрузку,
playback state, transport progress и завершение источника.

### Что делает пример

1. Создаёт [`runtime`](/ru/api#api-runtime).
2. Открывает [`playback_session`](/ru/api#api-playback).
3. Загружает локальный файл или URI.
4. Опросом читает [`playback_state`](/ru/api#api-playback), пока воспроизведение не завершится или не упадёт в ошибку.

### Ключевой код

```cpp
auto session_result = audio_runtime.open_playback_session();
if (!session_result) {
    std::cerr << session_result.error().message << '\n';
    return 1;
}

auto load_result = session_result.value().load(argv[1]);
if (!load_result) {
    std::cerr << load_result.error().message << '\n';
    return 1;
}

while (true) {
    const auto state = session_result.value().state();
    std::cout << "status=" << static_cast<int>(state.status)
              << " position_ms=" << state.position_ms
              << " duration_ms=" << state.duration_ms << '\n';

    if (state.status == sonotide::playback_status::idle ||
        state.status == sonotide::playback_status::error) {
        break;
    }

    std::this_thread::sleep_for(std::chrono::milliseconds(250));
}
```

### Почему он устроен именно так

- [`open_playback_session()`](/ru/api#api-runtime) показывает вход именно в
  high-level слой, а не только в render path.
- `load(...)` вынесен отдельно от создания сессии, чтобы ошибки загрузки были
  видны сразу и не смешивались с ошибками инициализации.
- Опрос [`playback_state`](/ru/api#api-playback) делает пример полезным и как
  CLI smoke test, и как основу для UI-интеграции.
- Таймаут в полном исходнике не даёт примеру зависнуть навсегда на проблемном источнике.

### Как адаптировать под свой код

Используйте этот шаблон, если нужно:

- строить player UI вокруг state snapshot-ов;
- переключать output device через `select_output_device(...)`;
- подключать встроенный EQ через `equalizer_state()` и методы
  [`playback_session`](/ru/api#api-playback).

### Запуск

```bat
build\msvc-x64-debug\examples\Debug\sonotide_playback_session.exe "C:\Windows\Media\notify.wav"
```

## Запуск из WSL

```bash
cmd.exe /c build\\msvc-x64-debug\\examples\\Debug\\sonotide_list_devices.exe
cmd.exe /c build\\msvc-x64-debug\\examples\\Debug\\sonotide_render_silence.exe
cmd.exe /c "build\\msvc-x64-debug\\examples\\Debug\\sonotide_playback_session.exe C:\\Windows\\Media\\notify.wav"
```

## С какого примера лучше начать

- Начинайте с `list_devices`, если нужно проверить границу runtime и метаданные endpoint-ов.
- Начинайте с `render_silence`, если у вас уже есть PCM и нужен только output path.
- Начинайте с `playback_session`, если хотите готовый playback workflow внутри фреймворка.
