## Why These Examples Matter

The command-line executables that ship with Sonotide are deliberate integration tests. They exercise specific layers of the framework in isolation:

Treat them as executable integration notes. They show the smallest possible safe entry point into each framework module.

## `sonotide_list_devices.exe`

Use this example when you want to verify that Sonotide can see the Windows
audio stack and return stable endpoint metadata.

### What the example does

1. Creates [`runtime`](/en/api#api-runtime).
2. Calls `enumerate_devices(device_direction::render)`.
3. Prints `id`, `friendly_name`, and the default-device flag.

### Key code

```cpp
auto runtime_result = sonotide::runtime::create();
if (!runtime_result) {
    std::cerr << runtime_result.error().message << '\n';
    return 1;
}

sonotide::runtime audio_runtime = std::move(runtime_result.value());
auto devices_result = audio_runtime.enumerate_devices(sonotide::device_direction::render);
if (!devices_result) {
    std::cerr << devices_result.error().message << '\n';
    return 1;
}

for (const auto& device : devices_result.value()) {
    std::cout << device.id << " | " << device.friendly_name;
    if (device.is_default_multimedia) {
        std::cout << " | default multimedia";
    }
    std::cout << '\n';
}
```

### Why it is written this way

- [`runtime::create()`](/en/api#api-runtime) is the framework boundary. If it
  fails, the example prints the structured error and stops immediately.
- [`device_info`](/en/api#api-device-model) is enough for application code. The
  example never reaches for COM objects.
- Printing the default-device flag makes the sample useful both as a smoke test
  and as a starting point for a settings UI.

### How to adapt it

Use this pattern if you need:

- an output-device picker;
- a diagnostics command that prints available endpoints;
- a first migration step away from direct `IMMDeviceEnumerator` usage.

### Run it

```bat
build\msvc-x64-debug\examples\Debug\sonotide_list_devices.exe
```

## `sonotide_render_silence.exe`

Use this example when you want to validate the output path without introducing
your own decoder, DSP chain, or test tone generation.

### What the example does

1. Creates [`runtime`](/en/api#api-runtime).
2. Builds a minimal [`render_callback`](/en/api#api-streams).
3. Opens a [`render_stream`](/en/api#api-streams) with default config.
4. Starts the stream and exits successfully if the critical render path works.

### Key code

```cpp
class silence_callback final : public sonotide::render_callback {
public:
    sonotide::result<void> on_render(
        sonotide::audio_buffer_view buffer,
        sonotide::stream_timestamp) override {
        std::fill(buffer.bytes.begin(), buffer.bytes.end(), std::byte{0});
        return sonotide::result<void>::success();
    }
};

sonotide::render_stream_config config;
auto stream_result = audio_runtime.open_render_stream(config, callback);
if (!stream_result) {
    std::cerr << stream_result.error().message << '\n';
    return 1;
}

auto start_result = stream_result.value().start();
if (!start_result) {
    std::cerr << start_result.error().message << '\n';
    return 1;
}
```

### Why it is written this way

- The callback writes zeros into [`audio_buffer_view`](/en/api#api-audio-format),
  which keeps the example deterministic and silent.
- [`render_stream_config`](/en/api#api-streams) uses the default path on
  purpose, so the sample is a clean validation of the framework defaults.
- A successful `start()` confirms that device selection, format negotiation,
  worker-thread setup, and the callback path all came up correctly.

### How to adapt it

This is the right starting point when your application already owns decoded PCM.
Replace the zero-fill block with code that copies your prepared frames into
`buffer.bytes`.

### Run it

```bat
build\msvc-x64-debug\examples\Debug\sonotide_render_silence.exe
```

## `sonotide_playback_session.exe`

Use this example when you want to see the high-level playback layer in action:
loading, playback state, transport progress, and source completion.

### What the example does

1. Creates [`runtime`](/en/api#api-runtime).
2. Opens [`playback_session`](/en/api#api-playback).
3. Loads a local file or URI.
4. Polls [`playback_state`](/en/api#api-playback) until playback completes or fails.

### Key code

```cpp
auto session_result = audio_runtime.open_playback_session();
if (!session_result) {
    std::cerr << session_result.error().message << '\n';
    return 1;
}

auto load_result = session_result.value().load(argv[1]);
if (!load_result) {
    std::cerr << load_result.error().message << '\n';
    return 1;
}

while (true) {
    const auto state = session_result.value().state();
    std::cout << "status=" << static_cast<int>(state.status)
              << " position_ms=" << state.position_ms
              << " duration_ms=" << state.duration_ms << '\n';

    if (state.status == sonotide::playback_status::idle ||
        state.status == sonotide::playback_status::error) {
        break;
    }

    std::this_thread::sleep_for(std::chrono::milliseconds(250));
}
```

### Why it is written this way

- [`open_playback_session()`](/en/api#api-runtime) demonstrates the high-level
  entry point instead of the lower-level render path.
- `load(...)` is separated from session creation so you can observe source-load
  failures clearly.
- Polling [`playback_state`](/en/api#api-playback) makes the example useful for
  both CLI verification and UI-oriented integration work.
- The timeout in the full example keeps the sample from hanging forever on a
  broken source.

### How to adapt it

Use this pattern when you want to:

- build a player UI around state snapshots;
- switch output devices through `select_output_device(...)`;
- integrate built-in EQ through `equalizer_state()` and the EQ methods on
  [`playback_session`](/en/api#api-playback).

### Run it

```bat
build\msvc-x64-debug\examples\Debug\sonotide_playback_session.exe "C:\Windows\Media\notify.wav"
```

## Running the examples from WSL

```bash
cmd.exe /c build\\msvc-x64-debug\\examples\\Debug\\sonotide_list_devices.exe
cmd.exe /c build\\msvc-x64-debug\\examples\\Debug\\sonotide_render_silence.exe
cmd.exe /c "build\\msvc-x64-debug\\examples\\Debug\\sonotide_playback_session.exe C:\\Windows\\Media\\notify.wav"
```

## Which example to start with

- Start with `list_devices` if you need to validate the runtime boundary and endpoint metadata.
- Start with `render_silence` if you already own PCM and only need an output path.
- Start with `playback_session` if you want a reusable playback workflow inside the framework.
