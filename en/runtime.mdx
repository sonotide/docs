title: "Runtime Context"
description: "How the runtime defines the framework boundary, enumerates devices, and acts as a factory for streams."
---

The [`runtime`](/en/api#api-runtime) is the foundational object in Sonotide. It transforms a collection of C++ headers into a living, breathing audio system. 

It holds ownership of the platform backend, exposes hardware device discovery, and acts as the factory for creating streams and playback sessions. If you're dropping Sonotide into an existing codebase, the `runtime` is the very first type you will interact with.

## 1. Initializing the Boundary

The [`runtime_options`](/en/api#api-runtime) structure defines what happens during the bootup phase:
- `initialize_media_foundation`: Bootstraps the MF platform (required for playback decoding).
- `enable_mmcss`: Enables Multimedia Class Scheduler Service to give worker threads real-time priority.

<Info>For 99% of applications, the default `{}` options are exactly what you need.</Info>

```cpp
auto runtime_result = sonotide::runtime::create();

if (!runtime_result) {
    // Log initialization failure
    return 1;
}

// Take ownership of the initialized runtime
sonotide::runtime runtime = std::move(runtime_result.value());
```

## 2. Discovering Hardware

Device enumeration is exposed safely via [`enumerate_devices(direction)`](/en/api#api-runtime) and [`default_device(direction, role)`](/en/api#api-runtime). 

Instead of dealing with opaque COM `IMMDevice` pointers, Sonotide returns clean, standard [`device_info`](/en/api#api-device-model) structs:

```cpp
auto devices_result = runtime.enumerate_devices(sonotide::device_direction::render);

if (devices_result) {
    for (const auto& device : devices_result.value()) {
        std::cout << "Found: " << device.friendly_name 
                  << " (ID: " << device.id << ")\n";
    }
}
```

## 3. Spawning Streams

Once the `runtime` is alive and you have a target device, you use the runtime factory methods to spawn audio sessions:

    Push generated or decoded PCM data to speakers/headphones.
  </Card>
  <Card title="open_capture_stream" icon="microphone">
    Pull recorded PCM data from a microphone or line-in.
  </Card>
  <Card title="open_loopback_stream" icon="laptop">
    Capture the system's output audio (e.g., recording desktop sounds).
  </Card>
  <Card title="open_playback_session" icon="play">
    Create a high-level session with a built-in decoder, volume, and EQ.
  </Card>
</CardGroup>

## When to Stop at Runtime

Should you use raw streams or jump straight to a `playback_session`? 

**Stay at the runtime/stream layer when:**
- Your application already handles its own decoding and just needs a sink.
- You are building a real-time synth or VOIP app.
- You are migrating away from an older raw WASAPI implementation step-by-step.

<Note>
If you want to play a `.mp3` or `.wav` file with pause, seek, and volume control, skip the raw streams and use a [`playback_session`](/en/api#api-playback) instead.
</Note>
