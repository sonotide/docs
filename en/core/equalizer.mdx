When using the `playback_session`, Sonotide provides a fully integrated equalizer with a dynamic band layout. You can keep the familiar default 10-band setup, reduce it to fewer bands, or build your own layout with up to 10 movable bands.

Since standard Windows WASAPI does not provide this out of the box, Sonotide wires it up natively inside the Media Foundation decode pipeline so you don't have to build your own DSP chain from scratch.

## 1. Equalizer Basics

To interact with the equalizer, use the methods directly attached to your `playback_session`:

```cpp
// 1. Turn it on
session.set_equalizer_enabled(true);

// 2. Load a built-in tuning curve
session.select_equalizer_preset(sonotide::equalizer_preset_id::rock);

// 3. Lower the output gain to prevent digital clipping
session.set_equalizer_output_gain(-1.5f);
```

## 2. Advanced Tuning

If the built-in presets (`rock`, `jazz`, `spoken_podcast`, etc.) don't fit your needs, you can still manipulate bands manually:

```cpp
// Boost one existing band
session.set_equalizer_band_gain(0, 3.5f);

// Move that same band to a different center frequency
session.set_equalizer_band_frequency(0, 72.0f);

// Change how narrow or wide the bell is
session.set_equalizer_band_q(0, 2.2f);

// Add a new band if you need a denser shape in the low mids
session.add_equalizer_band(450.0f, 1.5f);

// Remove a band later if the UI collapses back to a simpler layout
session.remove_equalizer_band(2);
```

<Warning>
  Pushing bands aggressively above `+0dB` can cause digital clipping. If you drastically boost certain frequencies, be sure to lower the `output_gain_db` to compensate.
</Warning>

## 3. Band Limits, Q Limits, and Default Layouts

The EQ is dynamic, but it is not unbounded. Sonotide exposes the limits explicitly so your UI can stay honest:

```cpp
const auto count_limits = sonotide::supported_equalizer_band_count_limits();
const auto frequency_limits = sonotide::supported_equalizer_frequency_limits();
const auto q_limits = sonotide::supported_equalizer_q_limits();

auto state = session.equalizer_state();
if (state.bands.size() < count_limits.max_band_count) {
    session.add_equalizer_band(2200.0f, 0.0f);
}

const auto range = session.equalizer_band_frequency_range(0);
if (range.has_value()) {
    session.set_equalizer_band_frequency(0, range->min_frequency_hz);
}

session.set_equalizer_band_q(0, q_limits.max_q_value);
```

If you want a ready-made layout, build one with `make_default_equalizer_bands(...)` and apply it through a full `equalizer_state`. Every default band starts with `default_equalizer_q_value`, which matches Sonotide's historical built-in bell width.

## 4. Reading the Equalizer State

You can take a snapshot of the current EQ settings via `session.equalizer_state()`. This returns a struct that contains everything your UI needs to draw faders:

## 5. Sampling the Exact Response Curve

If your UI draws an EQ curve, you no longer need to reimplement Sonotide's filter math on the frontend. Ask the framework for sampled response points instead:

```cpp
const std::array<float, 7> ui_frequencies_hz{
    31.5f, 63.0f, 125.0f, 500.0f, 1000.0f, 4000.0f, 12000.0f};

auto curve_result = session.sample_equalizer_response(ui_frequencies_hz);
if (!curve_result) {
    // If playback has not negotiated a format yet, call the free helper instead
    // and provide an explicit sample rate for your offline/editor scenario.
    auto state = session.equalizer_state();
    auto offline_curve_result = sonotide::sample_equalizer_response(
        state,
        48000.0f,
        ui_frequencies_hz);
}
```

This returns `equalizer_response_curve`, which already includes:

- the exact biquad response of every active band;
- the current per-band `q_value`;
- `headroom_compensation_db`;
- `output_gain_db`.

That makes Sonotide the single source of truth. The UI only needs to map `frequency_hz` to X and `response_db` to Y.

If the user is still editing a draft layout and you do not want to commit it to
the live session yet, use the preview API:

```cpp
sonotide::equalizer_preview_state preview_state;
preview_state.enabled = true;
preview_state.output_gain_db = 2.0f;
preview_state.bands = draft_bands;

auto preview_curve_result =
    session.preview_equalizer_response(preview_state, ui_frequencies_hz);
```

This preview path uses the same normalization and DSP math as the real EQ
pipeline, but it does not mutate `equalizer_state()`, the active preset, or
the playback graph.

## Design Note

The equalizer is built into the framework because it is an essential part of a reusable media playback pipeline. However, if your application needs complex parametrics (compressors, limiters, reverb), you should skip the built-in EQ and apply your own DSP chain manually on the raw chunks provided by a `render_stream`.
